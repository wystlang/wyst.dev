Wyst Compiler Roadmap

Updated: 2026-06-22

This roadmap tracks future compiler work and open exploration topics.

# Roadmap Rules

  1. Keep this file focused on future compiler work. Finished work belongs outside the roadmap.
  2. Require a concrete verification strategy before promoting any exploration topic into priority work or release planning. A systems capability is not done until a QEMU, binary-inspection, report-snapshot, or equivalent test proves why it exists.
  3. Preserve deterministic output as a tested compiler property. Build twice and compare bytes for every promoted verification target.
  4. Treat silent acceptance of unsupported syntax as a blocker for promotion or release.
  5. Keep version boundaries honest. Future work should not be presented as scheduled until it has been promoted from exploration into priority work or a release milestone.
  6. Do not use topic order as an implementation promise. When an exploration topic becomes concrete enough for priority work or release planning, create fresh scope with dependency order, non-goals, verification, and exit criteria.
  7. Use checked wyst design-contract fences for supported examples; mark future sketches with the future documentation contract defined in Appendix C.
  8. Keep design docs aligned with promoted scope. When promoted work changes a contract, report schema, target/artifact/debug policy, or resolves a design fork, update the source-of-truth registry and the relevant design doc.

# Scope Guardrails

Future promoted work must preserve these language and compiler boundaries:

  • Do not add compiler-exploitable undefined behavior.
  • Do not add hidden if-conversion, hidden autovectorization, hidden profile rewriting, hidden algorithm substitution, or hidden data-structure substitution.
  • Do not add hidden global allocation, hidden region inference, hidden garbage collection, hidden allocator replacement, or hidden cleanup control flow.
  • Do not add hidden automatic parallelization, hidden locking, hidden RCU-like rewrites, or hidden wait-free transformations.
  • Do not make firmware, OS, device, accelerator, PMU, debugger, editor, CI, SDK, remote service, or AI/generator behavior into core language semantics.
  • Do not make NUL-terminated strings the default Wyst text model.
  • Do not make textual macro/preprocessor semantics the main metaprogramming model.

# Priority Work Queue

Highest priority first. Items here are promoted roadmap scope: each owns its topic area, requires concrete verification before release planning, and should not be repeated under exploration.

  1. Target/artifact/linking foundation: promoted to release planning as the R8 / v0.8 milestone. Full scope, non-goals, dependency order, release targets, verification, and exit criteria now live in Release Milestones; this remains the highest-priority work and its scope is not repeated here.
  2. Diagnostics, conformance, and effect enforcement: write the required-diagnostics and conformance-test plan covering warnings, ABI edges, layout/order-sensitive behavior, public symbol contracts, #asm effect reports, function-pointer effect propagation, volatile retagging, and the compiler-ordering versus architectural-barrier boundary. (The exploration topic "Static mode/qualifier layer" would reframe this inferred-effect machinery as the first axis of a broader mode system; not promoted scope.)
  3. Reproducibility and lowering provenance: keep build optimization modes explicit and provenance-recorded, tighten schedule-hint wording for #schedule(throughput|latency), and clarify that ARM64 correspondence is recoverable through IR, explain, debug info, and provenance. (Translation validation against an IR reference semantics would strengthen this property; not promoted scope.)
  4. Production SMP support: implement explicit secondary startup, per-CPU initialization, runtime handoff, and QEMU/hardware validation stories.
  5. Thin standard library: after target/artifact/linking support can distribute components, design and ship allocation-explicit verified primitives and recipes without hidden runtime behavior.

# Release Milestones

Per the Tracking States, priority work that enters release planning graduates here with scope, non-goals, dependency order, numbered release targets, verification, exit criteria, and the design docs that will change. Completed releases move to CHANGELOG.md. v0.7 shipped the Standard Library Runtime And Data Contracts release (R7) through target 31-buffer-api-contracts; the next release is R8 / v0.8 and numbered targets resume at 32.

# Candidate R8: Target/Artifact/Linking Foundation

Confidence: committed. Promoted from Priority Work Queue item 1.

Theme: turn the relocation machinery the backend already computes for the final image into a real object/link toolchain — relocatable objects, external symbols, explicit binding/visibility, and deterministic multi-unit static linking — so "these units link into one inspectable image" is as explicit as the instruction-level semantics already are.

Scope:

  • Finalize explicit target facts and checked target descriptors as the carrier for object, relocation, symbol, and link policy, building on the existing #target(...) facts and named profiles (for example qemu-virt-aarch64-el2).
  • Relocatable object output: a wync -c / --emit-object mode that emits deterministic AArch64 ET_REL ELF, serializing the existing internal R_AARCH64_* relocation vocabulary into .rela.* sections with a real .symtab / .strtab, replacing the current "object-file output is not available" refusal.
  • Unresolved external AAPCS symbols: SHN_UNDEF references so a unit can call or read symbols it does not define, making gen-c-decls no-body declarations actually linkable rather than only describable.
  • Symbol binding and visibility model: STB_GLOBAL / STB_WEAK / STB_LOCAL and STV_DEFAULT / STV_HIDDEN with explicit source spelling, diagnostics, and symbol-table evidence (today only default binding exists).
  • Deterministic static linking: combine multiple objects plus the layout into a byte-reproducible ET_EXEC, with cross-object and external-symbol branch-veneer extension beyond the direct-symbol CALL26 / JUMP26 veneers the backend already synthesizes.
  • Debug-section relocation serialization: emit the debug-section relocations that already exist (for example R_AARCH64_PREL32) into the object, with DWARF content otherwise unchanged.
  • Archive/library distribution: a static-archive format for distributing compiled units (the existing release.wystar is a release-evidence bundle, not a linker archive; ar-compatible vs Wyst-native is an open fork below).
  • Foreign object linkage: link Wyst objects against externally produced AArch64 ELF objects (clang / gcc / as), at least as a binary-inspected smoke proof.
  • Artifact reports: a deterministic manifest describing target, relocations, defined/undefined symbols with binding/visibility, and link result for multi-unit images and foreign linkage.

Non-goals (each deferred to its own contract):

  • Dynamic linking, GOT/PLT, PIE/ET_DYN, shared objects, and ELF TLS (object-format/linker extensions topic).
  • Non-ELF object formats — this foundation is ELF-only, so PE/COFF (object-format/linker extensions) and the Mach-O artifact-mode proof (target-environments topic) are out.
  • init/fini-array and COMDAT/section-group handling (foreign-object compatibility; pairs with generics and monomorphization).
  • DWARF debug-information content beyond relocating the sections that already exist (debug/trace topic).
  • JIT/in-memory output, broad C/C++ interop, and the full AAPCS aggregate / variadic / vector surface (C-and-foreign-interface-widening topic).
  • Execution-environment profiles beyond the current bare-metal profile — Linux userspace, UEFI, EL1 freestanding (target-environments topic).

Dependency order: target descriptors -> relocatable object output (with binding/visibility and debug-reloc serialization) -> external symbols -> deterministic static linking with cross-object veneers -> archives and foreign object linkage -> artifact reports.

Release targets:

  • 32-relocatable-object-mode: wync -c emits one deterministic ET_REL per source file with .rela.*, .symtab / .strtab, and serialized debug-section relocations; proven by readelf / objdump snapshots and a build-twice byte compare. See the Target 32 sub-spec below.
  • 33-symbol-binding-visibility: global / weak / local and default / hidden binding with source spelling, accept/reject diagnostics, and symbol-table evidence.
  • 34-external-symbols: SHN_UNDEF external AAPCS references resolve at link time; gen-c-decls no-body declarations become linkable.
  • 35-static-link-multiunit: a project-graph fixture built as separate objects links deterministically into a byte-reproducible ET_EXEC, with cross-object veneers, and boots in QEMU with identical UART to the single-image build.
  • 36-foreign-object-linkage: a Wyst image links against an externally produced AArch64 ELF object; binary-inspected and booted in QEMU.
  • 37-archive-and-artifact-reports: static-archive distribution plus a deterministic artifact/link manifest reporting target, relocations, symbols, and link result.

Verification:

  • readelf -r / objdump snapshot tests on emitted objects (relocations, symbol binding, visibility).
  • Round-trip an existing QEMU virt fixture and the project-graph fixture as multiple objects -> linked ET_EXEC -> booted in QEMU with byte-identical expected UART.
  • External-linker cross-check: the same objects linked with system ld / lld as an ABI-well-formedness oracle (the shipped link path stays in-tree to keep determinism).
  • Build twice and compare bytes for every emitted object and linked image (Roadmap Rule 3).
  • No silent acceptance: unsupported object/link requests fail loudly with a diagnostic (Roadmap Rule 4).

Exit criteria:

  • wync -c emits deterministic, externally inspectable ET_REL objects whose relocations and symbols match the documented vocabulary.
  • A multi-object Wyst image links deterministically into a byte-reproducible ET_EXEC and boots in QEMU with output identical to the single-image build.
  • External-symbol references resolve against both Wyst and foreign AArch64 ELF objects; unresolved symbols fail loudly.
  • Symbol binding and visibility are source-spellable, diagnosed, and visible in the symbol table.
  • Every supported artifact mode reports target, relocation, symbol, and link status in a manifest; unsupported modes fail loudly or are documented non-goals.
  • Existing static-ELF / QEMU targets remain reproducible (no regression).

Design docs to update (Roadmap Rule 8):

  • source-of-truth.md — feature and schema status rows for each promoted artifact and interface surface.
  • chapter-16-object-format.md — ET_REL output, .rela.*, symbol table, binding/visibility, cross-object veneers, and the archive format (primary).
  • chapter-15-abi-spec.md — external AAPCS symbols and link-time ABI edges.
  • chapter-02-targets.md — target facts and descriptors for object/link policy.
  • chapter-03-project-builds.md — multi-unit object and archive builds.
  • chapter-23-debug-info.md — debug-section relocation serialization.
  • chapter-22-generated-manifest.md — artifact and link reports.

Open forks to settle during the milestone:

  • Archive format: ar-compatible vs Wyst-native, and how it relates to the existing release.wystar evidence bundle.
  • Whether the shipped linking path is wync's own deterministic linker or shells out to lld / ld (the determinism principle favors in-tree; external stays a verification oracle).
  • Visibility source spelling (for example #weak / #hidden) and its interaction with pub.
  • Cross-object veneer placement determinism.

Target 32 sub-spec: relocatable object output

Condensed contract for 32-relocatable-object-mode. Decisions are settled; the chapter-16 deltas land with the code per Rule 8.

  • CLI: wync -c / --emit-object replaces the current "object-file output is not available" refusal. Per-file translation units — each input .wyst file emits its own ET_REL object (output names derived from source basenames; -o requires a single input). Cross-file references (same module or imported) become SHN_UNDEF externals resolved at link time; sema still resolves the referenced declarations for types and ABI, but only the target file's definitions are emitted. -c is layout-independent and rejects --layout (placement is a link-time concern; no silent acceptance).
  • Shape: e_type = ET_REL, no program headers, e_entry = 0, sh_addr = 0; reuses the existing ELF writer, symbol, and string-table machinery.
  • Relocations: the executable path applies relocations, object mode emits them. The existing §5 vocabulary serializes to .rela.text / .rela.data / .rela.rodata / .rela.debug_* — ADR_PREL_PG_HI21 + ADD_ABS_LO12_NC and ADR_PREL_PG_HI21 + LDSTn_ABS_LO12_NC pairs split into two entries, plus CALL26, JUMP26, CONDBR19, ABS64, and PREL32. Wyst-internal offset kinds with no ELF equivalent (#tls / #percpu mrs+add) stay resolved at emit time, not emitted as relocations.
  • Symbols: referenced-but-undefined symbols become STB_GLOBAL / SHN_UNDEF entries (this makes gen-c-decls no-body declarations and cross-file references linkable). Relocations are defined-symbol-preferred, with a STT_SECTION + addend fallback for section-internal targets. Binding is unchanged (pub -> STB_GLOBAL, private -> STB_LOCAL); #weak / #hidden stay deferred to target 33.
  • Determinism: .rela entries sorted by r_offset (split pairs keep instruction order); undefined externals appended after defined globals in the existing §10 order; build-twice-and-compare-bytes applies to every emitted object.
  • Out of this target: multi-object linking and cross-object veneers (35), foreign-object consumption (36), archives (37), and #weak / #hidden spelling (33).
  • Chapter-16 landing (Rule 8, with the code): remove the "Per-module .o relocatable output" out-of-scope row in §9, add a Relocatable Object Output section, and extend §10 with the .rela ordering rules.

# Tracking States

Roadmap items use planning buckets. Language feature states live in design/source-of-truth.md, not in this file.

  • Priority work is promoted roadmap scope with actionable ordering. It may be split into release milestones only after scope, non-goals, verification, and exit criteria are written down.
  • Exploration topics are open design questions, possible features, or research threads. They are not release promises.
  • Promote an exploration topic into the priority queue only after it has clear user value, a syntax/API/workflow sketch, explicit non-goals, a design rationale, an implementation-gap assessment, and a concrete verification strategy.
  • When priority work becomes release planning, create a new release milestone with scope, target fixtures, exit criteria, and links to the design docs that will change. Also update the source-of-truth registry so readers can tell which future surfaces are normative and which remain experimental.

# Exploration Topics

These topics are planning memory only; each needs fresh design/compiler alignment before promotion. Do not duplicate priority work here; if promoted scope expands, move that text into the priority queue instead.

The list is ordered by importance, not by schedule. Per Roadmap Rule 6, topic order is not an implementation promise. Ordering weighs each topic against Wyst's stated identity and principles: make machine behavior visible, inspectable, and deterministic; performance transparency; reproducible lowering; direct ARM64 correspondence; tooling as a core language feature; and the verification-first working conventions. Topics that serve those principles directly, are foundational to other work, or have concrete byte-stable/snapshot verification rank higher. Topics that drift from the single-ISA, bare-metal systems core, brush the scope guardrails, or are large and speculative rank lower.

  • Object-format and linker extensions beyond deterministic static linking: dynamic linking, GOT/PLT relocation families, PIE/ET_DYN, shared objects, ELF TLS, COMDAT/section groups, init/fini arrays, and PE/COFF output. Symbol binding/visibility and cross-object/external-symbol branch veneers are promoted into Priority #1 and tracked there. (Each remaining item needs a language/linker contract before promotion.)
  • Explain and reporting surface: the wyst explain / report family, without hidden optimizer behavior, across three facets. (a) Performance/layout: static/profile/PMU/layout reports backed by a per-core microarchitectural cost-model database (latencies, port assignments, context-dependent critical-path vs loop-carried latency) with data-provenance fields (published / reverse-engineered / estimated), front-end vocabulary (decode pressure, code size, instruction-boundary regularity, uop-cache, condition-code dependencies), instruction-fusion and move-elimination visibility, and immediate-encodability diagnostics. (b) Measurement discipline: a wyst bench command distinct from wyst explain, line-level provenance tagging ([static] / [measured] / [compiler artifact]), a named static-vs-measured gap metric, and a rule that static estimates are never presented as measured performance. (c) Correctness/validation: surfaces showing whether a safety check dominates the operation it guards and whether the checked value is the value consumed, effect/authority boundaries for raw-pointer / trusted-writer APIs, and auditable address-range construction. (Directly serves the central performance-transparency and visible-behavior principles; correctness explainability aligns with the no-exploitable-UB stance and the verified-island methodology; requires explain/TMA-style, dominance, and accept/reject report snapshots and a versioned per-core descriptor with provenance. Dependency: the correctness/validation facet builds on the static mode/qualifier layer, and in-language PMU reporting depends on %read_pmu_counter from SIMD and machine-intrinsic expansion.)
  • Scheduling and PGO extensions beyond schedule-hint provenance: numeric branch weights, #schedule(parallel) loop dependency annotations, profile-guided allocation/scheduling, rematerialization beyond the current source-shaped constant/descriptor/address cases, arbitrary live-range splitting, and cross-function allocation. (Useful only if kept explicit and reproducible; must not become hidden optimizer behavior.)
  • Debug/trace/reload/visualization: trace/debug/watch/visualizer tooling outside ordinary source semantics, plus DWARF/source-floor expansion such as variable locations, type DIEs, inline expansion DIEs, CFI, fuller statement-row policy, Wyst-specific DWARF calling-convention constants, and remote trace transports. (Advances the machine-observability platform and visualization goals; requires snapshot verification and a DWARF debug-info contract.)
  • Toolchain trust/ecosystem beyond required diagnostics and conformance: reliability evidence, backend boundaries, provenance expansion, effect-taxonomy expansion, richer machine-readable diagnostic payloads, suppression policy, automation rules, and compiler phase-vocabulary cleanup (replacing internal and user-facing Phase N naming with capability-based backend, relocation, object, and report names). (Reproducibility and provenance are core principles; supports trust in every capability above; phase-vocabulary cleanup is low-risk honest-vocabulary hygiene per Roadmap Rule 6.)
  • Error-handling and partial-success model: errors as values with a defined standard error shape, representable partial success, allocation failure as an explicit handleable return, and no hidden control flow on error (no exceptions, no longjmp); explain can show error-propagation and allocation-failure paths. (Foundational language surface absent from current scope; must compose with the thin standard library and the no-hidden-cleanup guardrail; requires parser/sema/formatter and accept/reject snapshot coverage.)
  • Explicit arithmetic and numeric runtime forms: distinct source syntax for checked, saturating, flag-producing, and fixed-point arithmetic alongside the default wrapping operators, each lowering to its own inspectable sequence; explain distinguishes add from adds and shows NZCV flags produced and consumed; plus narrow runtime numeric primitives such as %fmod. (Extends the shipped wrapping default without adding hidden checks or optimizer license; each form is a byte-stable lowering with snapshot/disasm verification.)
  • Compile-time language surface: floating-point constant arithmetic, compile-time #if value/body selection, same-typed configuration variation, and the constant-evaluator/reporting rules needed to keep those forms deterministic. (Promote only with parser, formatter, sema, and snapshot coverage.)
  • Type and value surface expansion: post-core type and value language surface across four facets. (a) Type system: traits, type aliases, value parameters, defaults, inference, and generic constraints beyond the implemented type-parameter model. (b) Conversions and decay — two open language-design forks: a conversion taxonomy giving value-preserving numeric conversion, bit reinterpretation, and view/qualifier change each a distinct greppable form; and whether [N]T decays to @T at call boundaries, decided explicitly with recorded rationale and a diagnostic. (c) Aggregates: tuple parameters, nested or anonymous tuple policy, by-value fixed-array parameters, aggregate copy/ABI rules, and multi-return spill paths beyond the direct register model. (d) Enums and patterns: runtime tag extraction, explicit is lowering guarantees, tuple-payload enum variants, nested pattern policy, and source-visible tag/payload introspection. (Referential-stability decisions per the one-canonical-form principle; keep payload semantics explicit and reject inactive-payload reads; require ABI-visible and accept/reject snapshot verification.)
  • Static mode/qualifier layer: an erasable, inferred, opt-in-to-restrict qualifier system generalizing #deny and #noescape — escape (region-parameterized: stack frame vs caller-provided arena), unique/aliasing (use-after-free/use-after-move checking with the free still source-visible), and contention/portability (AXM data-race freedom for explicit SMP). Modes authorize but never silently perform reuse, free, or placement, and are never optimization license — the scope guardrails against hidden region inference, allocator replacement, cleanup control flow, and locking require exactly this fork. (Erasability is a byte-stable test: annotated vs unannotated source must lower identically; also needs accept/reject diagnostic snapshots and an explain modes report. Generalizes the Priority #2 effect machinery as "mode axis zero" and intersects Priority #3 provenance; adjacent to "Concurrency/progress/freshness", which it would subsume for the data-race-free case. Verified islands plus translation validation are a separate, heavier methodology thread. Dependency: the contention/portability (data-race-freedom) axis presupposes the concurrency/progress/freshness model and Priority #4 SMP, while the escape and unique axes have no such prerequisite and can proceed first.)
  • C and foreign-interface widening: a deterministic C header/importer workflow, pinned preprocessor/parser policy, refused-feature diagnostics, by-value C aggregate layout checks, function-pointer/variadic/vector/float import policy, and va_list compatibility. (Direct header import and broad C interop remain outside ordinary compilation.)
  • Library and collection expansion beyond the thin core: pointer-stable containers, unordered swap-back arrays, dense/sparse sets, explicit bounds-check contracts, broad collection APIs, generic container syntax, and optional allocator interfaces that do not hide allocation. (Keep allocation explicit across storage/typed-handle/buffer-style workflows.)
  • SIMD and machine-intrinsic expansion: vector comparison mask semantics, SVE/SVE2 predicate/vector surfaces, 128-bit pair atomics, floating-point atomics, checked #asm encoding expansion, %read_pmu_counter, full system-register table documentation, and optional preservation attributes such as #preserves(TPIDR_EL1). (All require direct ARM64 correspondence and effect/report coverage.)
  • Platform, tooling, documentation, and ecosystem beyond production SMP: physical Raspberry Pi validation, board run/copy/debug recipes, external GDB/LLDB/DAP adapter templates, editor/formatter/project workflow (range and project-wide formatting, in-place rewrite flags, project-aware catalog completion, richer quick fixes, Wyst-specific debug-adapter policy), checked documentation examples, teaching project walkthroughs, device protocol manifests, SDK discovery policy, generated-binding provenance, and external script/build-tool/editor-macro source generation. (These support Wyst users and verification evidence without becoming core language or compiler semantics; keep editor/formatter scope at the workflow layer. Dependency: board/hardware validation builds on the target environments and OS surfaces profiles.)
  • Target environments and OS surfaces: execution-environment profiles beyond the current bare-metal profile — AArch64 + Linux userspace, AArch64 + UEFI, and AArch64 + EL1 freestanding — with the syscall-only Apple Silicon Mach-O hello-world (_start, svc, rodata bytes, no external symbols, byte-stable binary-inspection/run verification) as the first concrete artifact-mode proof on a second format; plus the userspace/bare-metal device-driver workload (explicit MMIO Device attributes, DMA ownership, barriers with shareability domain, SVC as a cost-bearing event) and an OS I/O and memory-mapping surface (memory mapping with explicit lifetime / access width / failure semantics, OS file-facility facts in the target descriptor — O_DIRECT / F_NOCACHE / unavailable — source-visible alignment diagnostics instead of runtime EINVAL, sync-vs-async as an effect choice, and byte-offset parallel partitioning with explicit range ownership). (Extends Priority #1 target facts to execution environments; broad and partly OS-adjacent, so OS I/O must live in library/profile/effect surfaces, not the core; requires per-profile target descriptors and QEMU/binary-inspection verification.)
  • Concurrency/progress/freshness: explicit ownership, ordering, freshness, progress, and reclamation contracts. (Aligns with explicit memory behavior, but broad and guardrail-sensitive: must avoid hidden locking/RCU/wait-free rewrites. The data-race-freedom case may be served by the "Static mode/qualifier layer" topic via contention/portability modes.)
  • Cleanup/defer syntax: evaluate cleanup/defer syntax versus explicit control-flow patterns. (Language ergonomics; explicit control flow already covers the need, so lower marginal value; pairs with the error-handling model and the no-hidden-cleanup guardrail.)
  • Multi-architecture targets: narrow second-ISA proof with host/emitted target separation and vocabulary gates. (Speculative; tensions with the "direct ARM64 correspondence" identity.)
  • Self-hosting path: Wyst-built compiler helpers, bootstrap evidence, comparison harnesses, and architecture direction. (Largest and most speculative; depends on nearly everything above.)
