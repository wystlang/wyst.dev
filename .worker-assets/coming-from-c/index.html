Tradeoff guide

Coming from C

Wyst keeps direct memory and hardware access, but removes C's implicit promotions, hidden lowering, and optimizer license from undefined behavior.

Target: ARM64 only Safety: not memory-safe Runtime: no required libc
Familiar concept Wyst equivalent What you gain What you give up
Pointers and MMIO Typed address access such as u32@[addr] plus @mmio intent markers. Device access is visible in source and in reports. Invalid memory can still fault or misbehave; Wyst is not a sandbox.
Integer arithmetic Explicit widths and explicit casts. No implicit integer promotion and no signed-overflow UB for the compiler to exploit. More annotations where C would silently promote or convert.
Header and linker model Project manifests, modules, layouts, and deterministic AArch64 ELF output. The build graph and memory layout are compiler inputs rather than ambient host toolchain behavior. No drop-in compatibility with C build systems or libc.
Compiler optimizations Fixed, inspectable lowering and explicit optimization modes. The compiler will not rewrite around undefined behavior assumptions. You do not get a mature global optimizer for free.
sum_to comparison

The same loop from C.

C makes the loop compact, but signed overflow in C remains undefined. Wyst keeps the width and overflow behavior in the language contract so the compiler has no UB-powered rewrite license.

C

int sum_to(int n) {
    int acc = 0;
    for (int i = 0; i < n; i++) {
        acc += i;
    }
    return acc;
}

comparison snippet; signed overflow in C remains undefined

Wyst

#module demo.math

pub sum_to :: (n : i32) -> i32 {
    acc : i32 = 0
    i : i32 = 0
    while i < n {
        acc += i
        i += 1
    }
    return acc
}

Wyst comparison source; explicit width and visible control flow

Best fit

Bare-metal fixtures, small kernels, MMIO-heavy experiments, and code where inspectable lowering is more important than C ecosystem reach.

Main caveat

Wyst is not memory-safe: invalid memory can still fault or misbehave. The narrower claim is that the compiler will not exploit it for unrelated rewrites.

Current limit

ARM64 only, Rust bootstrap compiler, pre-1.0, and no C standard library replacement.
