Low-level programming you can actually read and reason about.

Wyst is an ARM64 bare-metal and kernel-oriented language and assembler. It sits between C and assembly: more expressive than raw AArch64, with none of the hidden magic of a C compiler. Memory ordering, source requirements, and overflow behavior are explicit in the source; semantic effects and post-lowering resource facts are surfaced for inspection. It does not make invalid memory safe: invalid memory access can still fault or misbehave.

Design principles

No magic. No surprises.

Wyst makes a single promise: what you write is what the machine does. The compiler's job is to lower it faithfully, not to outsmart you.

01
↻

Reproducible Lowering

The same source compiles to the same bytes, every time. Release tests build twice and compare the ELF byte-for-byte.

02
∎

Explicit Semantics

Data widths, arithmetic behavior, memory ordering, and effect boundaries are visible. Inferred effects are surfaced for inspection.

03
⊘

No UB-powered rewrites

The compiler does not use overflow, null, or out-of-bounds possibilities as optimization fuel. Invalid memory access can still fault or misbehave; Wyst's promise is that the compiler will not turn that into unrelated rewrites.

04
≈

Performance Transparency

Cost is visible at the source. No implicit vectorization, no hidden allocation, no global optimizer passes or profile-guided rewrites.

05
▦

Deterministic Output

Layout, symbols, and relocations are emitted deterministically. The ELF is a function of your source, not the machine that built it.

06
◎

Architecture-Aware

System registers, exception vectors, and AAPCS64 interop are first-class, close to the metal by design and not by escape hatch.

Non-goals

What Wyst deliberately leaves out.

Every omission below is a choice. A compiler that does less to your code is a compiler you can trust to do exactly what you asked.

✕ Hidden optimization passes Default lowering is fixed and inspectable. Non-default build optimization modes are opt-in and document their own reproducibility contract.
✕ Automatic algorithmic rewrites Your algorithm is your algorithm. The compiler won't swap your structure or loop shape for one it prefers.
✕ Implicit vectorization SIMD is written explicitly with fixed-width vector types, never conjured from a scalar loop.
✕ Hidden allocation semantics No allocation you didn't write. Storage is explicit, arena-first, with no mandatory runtime.
✕ Opaque compiler behavior Effects are inferred conservatively and surfaced; #deny lets you forbid the ones you don't want.
✕ Compiler-exploitable UB There's no undefined-behavior loophole to reason from. Defined in, defined out. Full stop.
What you give up

These are deliberate tradeoffs, not oversights. Knowing them up front is the point.

⇌ Implicit integer promotion Widths are explicit. You write the cast; the compiler never silently promotes a narrower type to a wider one. Expressions are more verbose as a result.
⇌ Portability across architectures Wyst targets ARM64 only. The ARM memory model, register classes, and barrier semantics are first-class, not abstracted over. An x86 or RISC-V port would be a different project.
⇌ A standard library There are no built-in containers, allocators, or I/O abstractions. Wyst provides the primitives; you compose the structures. Equivalent functionality is more code than idiomatic C++ or Rust.
Frequently asked

Straight answers.

What Wyst does today, where its limits genuinely are, and the lines it won't cross.

01

What can be built today?

Small ARM64 bare-metal and kernel-oriented programs: hand-rolled boot paths, MMIO experiments, simple runtime pieces, and fixtures that produce deterministic AArch64 ELF output. The compiler also supports wync check, fmt, project builds, LSP diagnostics, and explain reports.

02

Why isn't Wyst self-hosting?

The current compiler is a Rust bootstrap because the language and lowering contract are still changing. Self-hosting would be a major milestone, not proof of correctness by itself, and it is not required to validate the current AArch64 output path.

03

Why no LLVM backend?

LLVM is excellent when you want a mature optimizing backend. Wyst is testing a different contract: the lowering is meant to be inspectable, direct, and part of the language surface. There is no LLVM backend in the plan.

04

Is Wyst safer than C?

Not in the Rust sense. Wyst is not memory-safe and does not prevent bad pointers, bad lifetimes, or device misuse. The narrower safety property is that the compiler does not turn undefined behavior assumptions into unrelated rewrites.

05

What happens on invalid memory access?

Invalid memory access can still fault or misbehave. Wyst does not hide that behind a safety story. The point is that the compiler will not use the possibility of invalid memory access as fuel for an optimization that changes unrelated program behavior.

06

What should I not use Wyst for?

Do not use it for production kernels, security-critical code, portable user-space applications, or anything that needs a stable 1.0 ecosystem today. Use it for reading, experiments, fixtures, and understanding the language design.

07

Why ARM64 only?

A single ISA lets Wyst expose the real memory model, register classes, and barriers without inventing a portable abstraction over things that do not map cleanly between architectures. x86 or RISC-V would be a different project.

08

How does it sit between C and assembly?

Above assembly: structured control flow, named bindings, typed memory access, modules, explicit generics, enums, packed structs, and a register allocator. Below C: no implicit promotions, no hidden allocation, and no compiler-exploitable UB to reason from.

Project status

Built in the open. Early, but real.

Wyst is a serious hobby project in active development. The current v0.8-draft compiler, wync, is a zero-dependency Rust bootstrap with an LSP server, canonical formatter, project builds, and built-in lowering, effect, and storage reports, targeting ARM64 kernel and bare-metal experiments.

It's not finished. The ideas are real, the semantics are written down, and the output is deterministic.
