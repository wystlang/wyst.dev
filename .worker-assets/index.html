Low-level programming you can actually read and reason about.

Wyst is an ARM64 bare-metal and kernel-oriented language and assembler. It sits between C and assembly: more expressive than raw AArch64, with none of the hidden magic of a C compiler. Memory ordering, source requirements, and overflow behavior are explicit in the source; semantic effects and post-lowering resource facts are surfaced for inspection. It does not make invalid memory safe: invalid memory access can still fault or misbehave.

Design contract

No magic. No surprises.

Wyst makes the lowering contract part of the review surface: visible effects, deterministic output, explicit memory intent, and no UB-powered rewrites. Invalid memory access can still fault or misbehave; the compiler will not turn that into unrelated rewrites.

Reproducible lowering

The same source, layout, symbols, and relocations produce the same bytes.

Explicit semantics

Widths, arithmetic behavior, memory ordering, and effect boundaries are visible.

No UB-powered rewrites

Overflow, null, and out-of-bounds possibilities are not optimizer fuel.

Non-goals

What Wyst leaves out.

  • Hidden optimization passes: default lowering is fixed and inspectable.
  • Automatic algorithmic rewrites: your loop shape stays your loop shape.
  • Implicit vectorization: SIMD is explicit, never inferred from scalar loops.
  • Hidden allocation semantics: storage is explicit, with no mandatory runtime.
Tradeoffs

What you give up.

  • Implicit integer promotion: casts and widths are written out.
  • Portability across architectures: Wyst is ARM64 only.
  • A standard library: no built-in containers, allocators, or I/O abstractions.
  • Production readiness: do not use it for production kernels or security-critical code today.

What can be built today?

Small ARM64 bare-metal and kernel-oriented programs: boot paths, MMIO experiments, runtime pieces, and deterministic AArch64 ELF fixtures.

Why isn't Wyst self-hosting?

The current compiler is a Rust bootstrap because the language and lowering contract are still changing; self-hosting is a milestone, not proof of correctness.

Why no LLVM backend?

Wyst is testing a direct, inspectable lowering contract. LLVM maturity is useful, but there is no LLVM backend in this plan.

Is Wyst safer than C?

Not in the Rust sense. Wyst is not memory-safe; the narrower claim is that the compiler does not use undefined-behavior assumptions for unrelated rewrites.

What happens on invalid memory access?

Bad pointer behavior is still possible: invalid memory access can still fault or misbehave.

What should I not use Wyst for?

Do not use it for production kernels, security-critical code, portable user-space apps, or anything needing a stable 1.0 ecosystem today.

Project status

Built in the open. Early, but real.

Wyst is a serious hobby project in active development. The implementation lives in the open with docs, compiler tests, a formatter, an LSP server, project builds, and explain reports evolving together.

It's not finished. The ideas are real, the semantics are written down, and the output is deterministic.
