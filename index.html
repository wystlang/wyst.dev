Low-level programming you can actually read and reason about.

Wyst is a semantic ARM64 systems language and assembler. It sits between C and assembly: more expressive than raw AArch64, with none of the hidden magic of a C compiler. Memory ordering and overflow behavior are explicit in the source; machine effects are inferred from known primitives and surfaced for inspection.

Core principles

No magic. No surprises.

Wyst makes a single promise: what you write is what the machine does. The compiler's job is to lower it faithfully, not to outsmart you.

01
↻

Reproducible Lowering

The same source compiles to the same bytes, every time. Release tests build twice and compare the ELF byte-for-byte.

02
∎

Explicit Semantics

Data widths, arithmetic behavior, memory ordering, and effect boundaries are visible. Inferred effects are surfaced for inspection.

03
⊘

Zero UB Exploitation

There is no compiler-exploitable undefined behavior to mine for "optimizations." Defined inputs produce defined output.

04
≈

Performance Transparency

Cost is visible at the source. No implicit vectorization, no hidden allocation, no passes you didn't ask for.

05
▦

Deterministic Output

Layout, symbols, and relocations are emitted deterministically. The ELF is a function of your source, not the machine that built it.

06
◎

Architecture-Aware

System registers, exception vectors, and AAPCS64 interop are first-class, close to the metal by design and not by escape hatch.

Non-goals

What Wyst deliberately leaves out.

Every omission below is a choice. A compiler that does less to your code is a compiler you can trust to do exactly what you asked.

✕ Hidden optimization passes Default lowering is fixed and inspectable. Optimization is opt-in and documents its own reproducibility contract.
✕ Automatic algorithmic rewrites Your algorithm is your algorithm. The compiler won't swap your structure or loop shape for one it prefers.
✕ Implicit vectorization SIMD is written explicitly with fixed-width vector types, never conjured from a scalar loop.
✕ Hidden allocation semantics No allocation you didn't write. Storage is explicit, arena-first, with no mandatory runtime.
✕ Opaque compiler behavior Effects are inferred conservatively and surfaced; #deny lets you forbid the ones you don't want.
✕ Compiler-exploitable UB There's no undefined-behavior loophole to reason from. Defined in, defined out. Full stop.
What you give up

These are deliberate tradeoffs, not oversights. Knowing them up front is the point.

⇌ Implicit integer promotion Widths are explicit. You write the cast; the compiler never silently promotes a narrower type to a wider one. Expressions are more verbose as a result.
⇌ Portability across architectures Wyst targets ARM64 only. The ARM memory model, register classes, and barrier semantics are first-class, not abstracted over. An x86 or RISC-V port would be a different project.
⇌ A standard library There are no built-in containers, allocators, or I/O abstractions. Wyst provides the primitives; you compose the structures. Equivalent functionality is more code than idiomatic C++ or Rust.
Frequently asked

Straight answers.

What Wyst does today, where its limits genuinely are, and the lines it won't cross.

01

What does Wyst actually produce today?

Deterministic AArch64 ELF binaries for bare-metal and small-kernel work, tested under QEMU. The compiler, wync, is a zero-dependency Rust bootstrap with no LLVM or host linker in the emit path. It also ships an LSP server, a canonical formatter (wync fmt), project builds, and built-in lowering, effect, and storage reports.

02

How does it sit between C and assembly?

Above assembly: structured control flow, named bindings, typed memory access, modules, monomorphized generics, enums, packed structs, and a register allocator. Below C: no undefined behavior to exploit, no implicit promotions, no hidden allocation, no aliasing rules to trip over.

03

Why ARM64 only?

A single ISA lets Wyst expose the real memory model, register classes, and barriers without inventing a portable abstraction over things that don't map cleanly between architectures. x86 or RISC-V would be a different project.

04

What does "no compiler-exploitable UB" mean in practice?

Signed overflow wraps. Out-of-bounds reads return what the hardware returns. The compiler is never allowed to delete a branch on the grounds that the only way to reach it was undefined. #deny lets you forbid effects outright.

05

Is Wyst memory-safe?

No, and it doesn't claim to be. You can still dereference garbage. The promise is narrower: the compiler won't exploit the possibility that you did. That's a transparency property, not a safety one.

06

Will there be an LLVM backend?

No. The lowering is meant to be the spec. Routing through a backend that performs UB-based optimization would defeat the entire point of the language.

Project status

Built in the open. Early, but real.

Wyst is a serious hobby project in active development. The current v0.7 compiler, wync, is a zero-dependency Rust bootstrap with an LSP server, canonical formatter, project builds, and built-in lowering, effect, and storage reports, targeting ARM64 kernel and bare-metal experiments.

It's not finished. The ideas are real, the semantics are written down, and the output is deterministic.
