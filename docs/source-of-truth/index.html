Wyst Source Of Truth

Versioned authority, conflict resolution, feature states, and schema versions.

This document is the single versioned authority for Wyst language status, manual conflict resolution, and public schema version names. Detailed rules still live in the chapters and appendices; this page defines which rule wins when two documents disagree.

Current manual snapshot:

Surface Version Status Notes
Language wyst.language.v0.8-draft unreleased normative snapshot Covers the current main branch manual and compiler behavior after v0.7.
Native ABI wyst.nativeAbi.v0.8-draft unreleased normative snapshot Chapter 15 owns the detailed Native and AAPCS64 rules.
Report schema bundle wync.reports.v0 implemented Groups the report schemas listed below. Individual report payloads still carry their own schema field.
Interface schema bundle wync.interfaces.v0 implemented Groups CLI/editor/LSP adapter payloads listed below.

Released wync package version 0.7.0 implements the v0.7 release line. Rows marked implemented below refer to the current repository snapshot unless they explicitly name a released version.

# Authority Order

When prose, grammar, IR documentation, ABI rules, examples, or roadmap text conflict, use this order:

  1. This file wins for feature state, version names, authority rules, and whether material is implemented, future-version normative, experimental, reserved, deprecated, or removed.
  2. Chapter prose wins for user-visible language semantics in its scoped area. Chapter 15 wins for the Wyst Native ABI and AAPCS64 interop. Chapter 16 wins for emitted object/image format once an artifact mode is implemented.
  3. Appendix B wins for lexical grammar, parseability, disambiguation, and reserved tokens. A grammar production does not make a feature available unless this registry or the owning chapter marks it implemented or future-version normative.
  4. Appendix A wins only for compiler-internal IR shape and verifier invariants. If Appendix A conflicts with user language semantics or ABI rules, the chapter rule wins and Appendix A must be corrected.
  5. Appendix C controls example categories. Examples never create language semantics by themselves.
  6. roadmap.md is planning material. It may propose or promote work, but it does not override this file or a chapter. A roadmap milestone becomes future-version normative only when this file marks that surface future-version normative.
  7. Compiler tests and snapshots are implementation evidence. If tests and docs disagree, treat it as a bug in one of them and update this file plus the owning chapter or implementation together.

# Feature States

Every feature row uses one of these states:

State Meaning
Implemented Accepted by the current compiler or emitted by the current tooling, with tests or snapshots protecting the contract.
Future-version normative The rule is already chosen for a named future version or release target, but the current compiler may reject it or omit it.
Experimental Planning or research material only. It is not normative, not scheduled, and not a compatibility promise.
Reserved Syntax, names, encodings, or namespaces are held so future additions remain non-breaking. Reserved forms are rejected unless a later row changes their state.
Deprecated Still accepted, but discouraged and scheduled for replacement or removal. A deprecated row must name the replacement or removal condition.
Removed No longer part of Wyst. Implementations should reject it or treat it as an unknown construct; examples must not use it except in diagnostics.

Features not listed here inherit the state of the nearest listed feature family. A chapter that introduces a new externally visible syntax form, ABI rule, report field, CLI payload, object artifact, editor behavior, or roadmap milestone must add or update a row in this registry.
