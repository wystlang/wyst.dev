Chapter 13

Chapter 13: Wyst Scheduling Semantics

Scheduling regions, deterministic reordering boundaries, and target-aware scheduling modes.

Canonical scope. The implicit schedule.default mode, the #schedule(strict | relaxed | throughput | latency) directive, the ordering-constraint vs optimization-hint distinction, and reproducibility scope. IR-level region representation of schedule blocks lives in appendix-a-ir.md §4.3.

# Semantic Boundary

Scheduling modes are permissions for instruction ordering, not a way to change source meaning. Optimization and IR rules build on the source evaluation order, effects, memory ordering, and dependency rules established here. Scheduling may move pure work only when the move is observationally equivalent under those rules.

# Scheduling Semantics

Wyst assigns every emitted instruction to a named scheduling mode. Code that is not enclosed by an explicit #schedule(...) directive is in the distinct implicit mode schedule.default. Wyst also exposes scheduling permissions explicitly via the #schedule directive. Modes are divided into two categories:

#schedule modes are source-level scheduling modes, not build optimization modes. Build optimization modes are selected separately with --optimization or optimization = "..." in wyst.project (see chapter-17-optimization-modes.md). Whatever build optimization mode is selected must still respect the #schedule regions in the source.

Reports and generated build identities must record the selected scheduling modes whenever a mode can affect generated bytes. The implicit mode is reported as schedule.default, not as an omitted policy.

Ordering constraints — specify what the assembler is and is not permitted to do. Output is reproducible within a compiler version, target, and #schedule mode.

Optimization hints — express intent; the assembler may use them to improve code but the exact output is implementation-defined and may change across compiler versions or microarchitecture targets. For a fixed source input manifest, compiler binary, build optimization mode, target, and #schedule mode, the chosen output must still be deterministic across invocations. Do not use hint modes where source-order layout, exact instruction count, or cross-version instruction order must remain stable.

# Checked Scheduling Example

#schedule(strict) is valid inside an ordinary function body:

#module scheduling_demo

step :: (a : u64, b : u64) -> u64 {
  #schedule(strict) {
    sum : u64 = a + b
    return sum
  }
}

# Default Scheduling

Ordinary code without a #schedule directive is compiled under the named mode schedule.default.

schedule.default is a distinct mode. It is not an alias for #schedule(strict), because it permits deterministic reordering of pure work. It is also not an alias for #schedule(relaxed), because reports, IR region identity, and build identity record it as schedule.default. Its pure instruction-ordering permissions are intentionally the same as the relaxed mode unless a later language version changes this row.

The default mode's permissions are:

  • Semantic dependencies: an instruction may move only after every value, control, address, and storage dependency it reads has been produced. An instruction that produces a value must remain before every use of that value.
  • Evaluation order: source expression evaluation remains the left-to-right contract from Chapter 7. Default scheduling may change only target instruction order after typed IR dependencies prove the same observable result.
  • Effects: calls, inline assembly, system-register operations, traps, compiler barriers, volatile accesses, atomics, stores, and other effectful operations are scheduling barriers. Pure instructions must not cross an effect barrier when that would change the order in which effects are observed or reported.
  • Alias proofs: ordinary memory accesses are not treated as independent by default merely because their source expressions differ. Reordering memory operations requires a closed alias proof from typed address, storage, and memory-model facts. Without such a proof, memory operations remain in source order relative to other memory and effect operations.
  • Memory model: default scheduling must preserve volatile order, atomic order, release/acquire and seq_cst constraints, barrier synchronization, per-location modification order, and happens-before edges from chapter-09-memory-model.md. It never inserts a hidden hardware barrier and never weakens an explicit one.
  • Deterministic tie-breaking: when multiple pure instructions are ready, the compiler ranks them by dependency class, then by their canonical IR/source order. Equal-ranked ready instructions keep source order. No hash iteration, host timing, thread interleaving, or environment-dependent order may decide a tie.
  • Target instruction selection: the target backend may select any target-legal instruction form that preserves the selected order, effects, memory constraints, ABI facts, and source-visible rounding/trap behavior. Target-specific lowering must use deterministic target descriptor facts and must not reinterpret schedule.default as a microarchitecture hint mode.

Use ordinary code when deterministic dependency-safe pure reordering is acceptable and no explicit scheduling boundary is needed. Use #schedule(strict) when source-level semantic operation order or inspection of source-level intermediates must be preserved.

# Strict Scheduling

#schedule(strict) {
    raw  := u64@[a]
    next := raw + 1
}

#schedule(strict) is defined at the level of source-level semantic operations. A semantic operation is a source-visible evaluation step such as a runtime arithmetic operation, load, store, call, intrinsic, inline assembly operation, trap, branch condition, or source-declared temporary whose value is part of the declared inspection contract. Strict scheduling is not a one-source-line-to-one-instruction mapping, and it is not a promise that target machine instructions appear one-for-one with source lines.

Ordering constraint. Source-level semantic operations inside the strict region remain in source evaluation order. Lowering may introduce required support instructions, but it may not move, duplicate, combine, or eliminate semantic operations when doing so changes an observable intermediate value or the declared inspection contract. Non-dependent semantic operations are still sequenced by their source order.

The braces of a #schedule(strict) block are also hard scheduling boundaries for semantic operations. Operations outside the block must not be moved into the block, and operations inside the block must not be moved out of it. The boundary is a compiler scheduling fence only; it does not emit a hardware barrier and does not change the CPU memory-ordering rules described in chapter-09-memory-model.md. Use %compiler_barrier() when a statement-level compiler memory fence is needed without placing the surrounding operations inside a strict schedule region.

Strict mode permits the following target-lowering details when they preserve the semantic-operation sequence and any declared inspection contract:

Lowering detail Strict-mode rule
constant folding Permitted for compile-time-constant expressions and other pure values whose folded result is the same source-level value and whose intermediate steps are not part of the inspection contract.
folded address modes Permitted when folding an address calculation into a load/store addressing mode does not remove a source-declared address value that must remain inspectable. The load or store itself stays in source order.
support instructions Permitted before, after, or around a semantic operation when required to materialize immediates, addresses, register moves, masks, extensions, or other target operands.
register spills and reloads Permitted as allocator support instructions. They are not source semantic operations, but they must not reorder source loads, stores, calls, volatile operations, or inspectable temporaries.
prologues and epilogues Permitted as ABI/frame support outside the source semantic sequence for a function body. Strict regions do not suppress required frame setup or teardown.
ABI marshalling Permitted for parameter passing, return-value handling, aggregate copies, indirect returns, and call-preserved register handling. Calls remain semantic barriers in source order.
dead pure temporary removal Permitted only when the temporary has no effect, no use, no source-visible address or storage identity, and no declared inspection contract.
instruction combination Permitted only when the combined instruction implements one semantic operation or combines support instructions without changing any observable intermediate. Combining distinct source-level semantic operations into one result is not permitted when it changes an intermediate value or inspection contract.
vector-slot allocation changes Permitted as generated-resource allocation. Slot placement and register choice are not semantic operations, but they remain deterministic and must satisfy any frame, vector-slot, ABI, and report contracts.

Floating-point contraction is governed by chapter-07-operators.md: %fma is the only Wyst source operation that requests fused multiply-add/subtract lowering. If the source does not use %fma, no #schedule mode may fuse a * b + c into a single-rounded multiply-add. #schedule(strict) is not needed to prevent that transformation.

Use #schedule(strict) when:

  • source order is semantically significant
  • source-level intermediate values must remain inspectable
  • scheduler reordering must be disabled inside a bounded region
  • timing-sensitive sequences must not be reordered

Strict mode does not by itself freeze exact instruction count, exact register choice, spill placement, prologue shape, or ABI marshalling. Use frame constraints, slot-budget diagnostics, disassembly snapshots, and generated reports when those post-lowering resources are part of the contract.

# Exact Code Contracts

Exception vectors, counted sequences, timing-sensitive operations, and byte identity boundaries use #exact(...) on a function or label when strict scheduling is not strong enough. #exact is a post-lowering artifact contract, not a source-level scheduling mode. It does not authorize reordering and it does not change source meaning; it constrains what the backend is allowed to emit for that code item.

#module exact_code_demo

tick :: () #exact(instructions = 2, families = "hint,branch", bytes = "1f2003d5 c0035fd6", prologue = "absent", spills = 0, veneers = 0, section = ".text", align = 4) {
  %nop()
}

The supported exact-code fields are:

Field Contract
instructions = N the emitted body has exactly N AArch64 instructions
families = "a,b" every emitted instruction belongs to one of the listed families: arithmetic, branch, compare, hint, load-store, logical, move, system, adr, or other
bytes = "hex" the emitted body bytes exactly match the hexadecimal byte string, ignoring spaces and underscores in the source string
registers = "name:xN,other:vM" named parameters or locals are assigned exactly to the listed registers after allocation
prologue = "present" / "absent" the function or label either has or does not have compiler-generated frame setup
spills = N register allocation produces exactly N spill slots
veneers = N veneer planning inserts exactly N veneers for the text chunk containing the code item
section = ".name" the code item is emitted into exactly that executable section
align = N the emitted file and address offsets are aligned to N bytes

The compiler verifies #exact after ABI lowering, register allocation, frame layout, target instruction selection, veneer planning, and text emission. If a contract cannot be satisfied, the build is rejected with a diagnostic that names the failed requirement and the extra instruction, byte, register, spill, veneer, prologue, or placement resource that would be required. The compiler must not silently insert support code to make an exact-code region work; the programmer must relax the contract or change the source.

Use #schedule(strict) for ordinary source-operation ordering. Use #exact only when the post-lowering artifact itself is the contract. Keeping the two facilities separate prevents strict scheduling from becoming a byte-freezing mode for ordinary code.

# Relaxed Scheduling

#schedule(relaxed) {
    lhs := u64@[a]
    rhs := u64@[b]
    sum := lhs + rhs
}

Ordering constraint. The assembler may reorder instructions that have no data dependency between them. An instruction may not be moved past any instruction it depends on, and may not be moved past any instruction that depends on it. Data dependencies are respected strictly.

Output is a pure function of source, compiler version, and target triple: the same source compiled with the same compiler version against the same target always produces the same instruction order, across all invocations. This is a specified guarantee, not an implementation convenience. Instruction order is not guaranteed to be preserved across compiler versions.

Use #schedule(relaxed) when dependency ordering is sufficient and small reorderings for pipeline efficiency are acceptable.

# Throughput Scheduling

#schedule(throughput) {
    sum = a + b
}

Optimization hint. The assembler attempts to schedule instructions to maximize throughput — preferring sequences that keep multiple execution units busy. The exact output is implementation-defined and may change across compiler versions or microarchitecture targets. Within one compiler binary and the same build inputs, the chosen instruction order is deterministic.

Do not use #schedule(throughput) in #ventry bodies where instruction count must be predictable or in code that requires cross-version stable instruction order. Floating-point rounding and multiply-add contraction are not controlled by scheduling hints; if the source does not use %fma, the compiler does not fuse.

# Latency Scheduling

#schedule(latency) {
    chain += next
}

Optimization hint. The assembler attempts to schedule instructions to minimize critical-path latency — preferring sequences that reduce dependency chain depth. The exact output is implementation-defined and may change across compiler versions or microarchitecture targets. Within one compiler binary and the same build inputs, the chosen instruction order is deterministic.

Same caveats as #schedule(throughput). Use only in non-critical paths where the hint mode's cross-version instruction-order instability is acceptable in exchange for potential latency improvement.

# Layout Constraint

Hint scheduling modes must not change code layout.

#schedule(throughput) and #schedule(latency) may reorder instructions within a single basic block only. They must not:

  • move instructions across basic block boundaries
  • change the ordering of basic blocks in the emitted binary
  • change which section a function or block is placed in
  • alter the control flow graph shape (branch targets, fall-through targets)
// The hint mode may reorder the two independent loads,
// but may NOT move them past the branch or change which
// block is the fall-through target.

#schedule(throughput) {
    a := u64@[p]
    b := u64@[q]
    if a > b {
        hot_path()
    } else {
        cold_path()
    }
}

Rationale. Code layout — block ordering, section placement, fall-through direction — is determined by the source structure and explicit directives (#likely/#unlikely, #cold, #section), never by scheduling hints. Separating layout from scheduling preserves two properties:

  1. Reproducibility scope. Layout decisions are deterministic under the Reproducibility Model's input catalog (same source input manifest, compiler version, build optimization mode, target, and #schedule modes → same layout). Hint modes affect only instruction order within blocks, which is deterministic for the same compiler binary but may evolve across compiler versions or microarchitecture targets.

  2. Composability. A programmer can pin layout with #likely/#unlikely and let a hint mode optimise instruction order within each block without the two interacting. See chapter-08-functions.md §2.7.2 for branch hints and chapter-04-modules.md for hot/cold section conventions.
